# P3 ‚Äî transcript‚Üîframe alignment (anchored-slide selection)

**Status:** ‚ùå **experiments run (2026-07-13) ‚Äî ALL THREE DIRECTIONS FAILED THE GATE.
Nothing built. Recommendation: ship the P1+P2 feature as-is; do not pursue P3.**
**Scope:** local/desktop only ‚Äî never web. **Builds on:** [`FRAMES.md`](FRAMES.md).

## Experiment results ‚Äî three directions, all gated, all negative

Method (mirrors P1/P2): scripted prototypes in the scratchpad, real digests via the
shipped `generateDigest` + clamp, measured before touching product code.

**A. Salience-weighted selection** ‚Äî pick frames by detail √ó narration specificity at
their timestamp (¬±8s window). Tested on 2 videos (paywalls + onboarding), same dense
candidate pool, detail-only vs salience-weighted, full digest each.
‚Üí **FAIL.** Changed only **4 of 24 frames** on both videos; digests statistically
indistinguishable (paywalls 23 vs 22 numeric tokens; onboarding 45 vs 49; anchored
specifics ‚Äî Opal 7‚Üí17%, Slopes 25%, Headspace 10%, Brian Chesky‚Ä¶ ‚Äî captured equally).
*Why:* detail-max already lands on anchored slides (a video shows a stat slide *while*
discussing it), so salience adds nothing.

**B. Per-frame transcript context** ‚Äî annotate each frame with its timestamp + the
narration around it ("frame-07 @4:32, on screen while: '‚Ä¶'") so the model attaches its
on-screen numbers to the right point. Bag-of-frames vs contextualized, same 24 frames.
‚Üí **FAIL, and counterproductive.** Same specifics count (55 vs 51 numeric) but
qualitatively *worse*: bag captured on-screen detail (Blinkist's Day-0/Day-7 timeline,
Slopes' "$39" pricing, Tipstop's A/B) that the contextualized version *missed*. *Why:*
handing the model the narration-at-that-moment makes it **paraphrase the narration**
instead of **reading fresh detail off the frame**. Context suppresses extraction.

**Density (scene vs dense candidate pool)** ‚Äî the lever the A/B experiments seemed to
point at. Shipped scene-detection pool vs a dense interval pool (~1 frame/6s), both
‚Üí detail-select 24 ‚Üí digest with the shipped clamp.
‚Üí **MARGINAL / WASH.** Dense was slightly richer (34 vs 27 total numeric tokens, longer
digest) but *not* systematically better at high-value frame specifics: both captured the
same count of distinct percentages (8 each) and ~3 frame-only percentages each ‚Äî just
*different* ones depending on which frames landed. The extra dense tokens are mostly
incidental numbers (day counts, spelled-out prices), within run-to-run noise. Not worth
the added extraction cost. (Also: approach C, two-pass LLM claim-anchoring, is untested
and costs 2√ó LLM calls ‚Äî not worth trying given A/B/density all flat.)

## Conclusion

The shipped design ‚Äî **bag-of-frames + numbers-encouraged/names-strict clamp on a
reasonable candidate pool** ‚Äî is at the sweet spot. Frame value comes from *reading the
frames*, and the model already does that well; alignment, context-injection, and denser
sampling each add complexity for negligible or negative return. **Ship P1+P2 as-is;
P3 is closed.** The gate-first discipline prevented building three separate dead-ends.

## The gap P3 closes

P2 made selection pick the most **visually detailed** frames (blanks and talking-heads
dropped). But the E2E proved *detailed ‚âÝ valuable*: on the paywalls video the 24
selected frames were mostly **incidental B-roll** ‚Äî paywall screenshots the video
flashes by (Arrow Gold $29.99, L√≥vi Premium‚Ä¶). The digest correctly ignored them,
because **the narrator wasn't talking about them.**

The frames that actually paid off were the ones whose on-screen content **anchored to a
point the narration was making** ‚Äî the spin-wheel `75%` (narrator discusses spin-wheel
tactics), the Blinkist stats in the manual test (narrator discusses Blinkist's
redesign). A visual-detail score can't tell an anchored stat slide from busy B-roll;
they're all just "detailed slides."

**P3 = prefer frames whose timestamp lands on substantive, specific narration, and tell
the digest which narration each frame belongs to.**

## The enabler (already present, currently discarded)

Both sides already carry timestamps ‚Äî the join just isn't used:
- **Frames:** `frames.js extractFrames` returns `[{ path, offsetSec }]` (P1 built the
  `showinfo`/`pts_time` timestamping).
- **Transcript:** `transcript.js fetchTranscript` returns `[{ text, offset }]` (seconds).

But the **timestamps are thrown away before the digest**: the frontend flattens
segments to plain text via `buildPlainTranscript()` (drops offsets), and
`POST /api/digest` receives only `text`. So today the digest sees a loose pile of
frames and a timeless wall of text. P3's precondition is keeping the segment offsets
alive down to the frame-selection + digest step.

## Approaches (cheapest first)

### A. Salience-weighted co-selection *(recommended for P3 v1)*
Re-rank candidate frames by **both** visual detail **and** narration salience at their
timestamp, instead of detail alone.
- **Salience of a transcript window** near a frame's `offsetSec` = density of *specific*
  tokens: numbers, currency, `%`, dates, and capitalized proper nouns/multi-word names
  (a cheap regex/heuristic ‚Äî no LLM). A frame sitting on "‚Ä¶Blinkist redesigned its trial,
  result +23%‚Ä¶" scores high; one over "‚Ä¶so anyway, you know, the thing is‚Ä¶" scores low.
- **Combined score** = `norm(jpegSize) * (1 + w * salience)` ‚Äî keep the P2 detail proxy,
  boost frames aligned to specific narration. Per-temporal-bin max by combined score
  (reuse P2's `pickByBins`).
- Zero extra LLM cost; pure heuristic over data we already have.

### B. Per-frame narration context in the prompt *(pairs with A, recommended)*
Instead of "here are 24 frames," tell the digest **when** each frame was on screen and
what was being said: `frame-007.jpg (~4:32) ‚Äî on screen while: "<the transcript window
around 4:32>"`. This lets the model fuse a frame's numbers into the *right* point rather
than guessing where a loose frame belongs. Changes `buildFramePromptBlock` to emit a
timestamped, context-annotated frame list. Cost: a bit more prompt text.

### C. Two-pass claim-anchoring *(P3.5 ‚Äî only if A+B underperform)*
A cheap first LLM pass extracts the video's key claims + their transcript timestamps;
then select and read *only* the frames at those timestamps to ground them. Most powerful,
most expensive (2√ó LLM calls, more orchestration). Defer until A+B are measured.

**Recommendation:** ship **A + B** together for P3 v1 ‚Äî they share the one precondition
(segment offsets reaching the frame path), add no LLM cost, and directly target the gap.

## Data-flow change (the one real plumbing cost)

Segment offsets must reach frame selection + the digest. Two options:
1. **Frontend sends `segments`** (it already holds `lastSegments`) alongside `text` on
   `POST /api/digest`. No redundant fetch; but a bigger request body (fine ‚Äî
   local/desktop only, and the web cap doesn't apply since frames are web-disabled).
2. **Server re-fetches** `fetchTranscript(videoId)` for its offsets when
   `includeVisuals`. No client change; costs one keyless transcript re-fetch.

Lean **option 1** (client already has the segments; avoids a duplicate fetch). Guard:
segments are only needed on the frames path, so this stays inside the `wantVisuals`
branch and never touches the plain text-digest contract.

## Code touchpoints

- **`frames.js`** ‚Äî `extractFrames` should return a **larger candidate pool** (e.g. keep
  the top ~2‚Äì3√ó `maxFrames` by detail, or return all candidates with `{path, offsetSec,
  size}`) so the aligner can re-rank by salience instead of the visual-only cap deciding
  first. Move the final `maxFrames` cap into the aligner. `selectDetailFrames` stays as
  the fallback when no segments are supplied.
- **new `alignFrames.js`** (or a function in `digest.js`) ‚Äî `selectAnchoredFrames(candidates,
  segments, maxFrames, opts)`: compute per-frame salience from the nearest segment
  window, combine with detail, per-bin max, cap, and (for approach B) return each kept
  frame with its `contextText`. Pure + unit-testable (feed synthetic candidates+segments).
- **`digest.js`** ‚Äî `generateDigest` accepts `opts.frames.items[].contextText` and
  `buildFramePromptBlock` emits the timestamped, context-annotated list (approach B).
- **`server.js` `/api/digest`** ‚Äî accept `segments`; when `wantVisuals`, pass them into
  the align step; keep the best-effort/degrade + `finally` cleanup exactly as P2.
- **`public/index.html`** ‚Äî send `segments` (from `lastSegments`) in the `includeVisuals`
  request. Local/desktop only, as today.

## Faithfulness

Keep P2's **numbers-encouraged / names-strict** clamp verbatim ‚Äî approach B *reduces*
fabrication risk (it hands the model the real narration context, so it has less reason to
invent). Do not relax the names rule; the alignment context is grounding, not licence.

## The experiment to run FIRST (gate)

Mirror the P1/P2 method ‚Äî measure before building:
1. Take 2 videos: the Mobbin paywalls teardown (have it) + one lecture/slide talk where
   visuals carry info (e.g. a 3Blue1Brown-style or conference-slides video).
2. For each: run the P2 digest (detail-only selection) vs a **prototype** salience-weighted
   selection (approach A, scripted in scratchpad ‚Äî no need to wire the server yet).
3. Compare: does salience-weighting surface the **narration-anchored** stat slides
   (Opal 7‚Üí17%, Slopes 25%, Blinkist +23/-55) that P2 missed? Count frame-only specifics
   captured, and check the selected frames' contents (ground-truth via an image agent, as
   in the prior tests).
4. **Ship P3 only if salience-weighting measurably captures more anchored specifics than
   detail-only.** If detail-only is already close, P3 isn't worth the plumbing.

## Non-goals / risks

- **Not web.** Same as frames overall.
- **Timestamp skew.** A frame's `offsetSec` (scene-onset) may lead/lag the narration that
  references it by a few seconds; the salience window must be generous (¬±~8s), and the
  onset-vs-mention gap is a known imprecision to measure, not assume away.
- **Salience heuristic is crude.** Regex specificity ‚âÝ semantic importance; it will miss
  claims phrased plainly and over-weight number-dense filler. Approach C (LLM claim
  extraction) is the fallback if the heuristic is too noisy ‚Äî but it's more cost, so gate
  on the experiment.
- **Diminishing returns.** P1+P2 already deliver measured value (‚âà2√ó concrete figures).
  P3 is the "surface the *best* on-screen facts, reliably" upgrade ‚Äî real, but not a
  prerequisite for the feature being useful.

## Open decisions

2. Salience weight `w` and window size (¬±8s assumed) ‚Äî tune in the experiment.
3. Candidate-pool size returned by `extractFrames` (2√ó vs 3√ó `maxFrames`).
4. Whether approach B's per-frame context is worth the added prompt length on its own, or
