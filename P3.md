# P3 — transcript↔frame alignment (anchored-slide selection)

**Status:** ❌ **experiment run (2026-07-13) — GATE FAILED for approach A. Not built,
not recommended as specced.** **Scope:** local/desktop only — never web.
**Builds on:** [`FRAMES.md`](FRAMES.md) (P1+P2, shipped).

## Experiment result — approach A does NOT clear the gate

Ran the gating experiment (see the last section) on two videos (Mobbin paywalls + Mobbin
onboarding). For each: the same dense candidate pool (~100–125 frames @ 1 per 6s), two
selections — **detail-only (P2)** vs **salience-weighted (P3-A: detail × narration
specificity in a ±8s window)** — then a full digest per set with the shipped clamp.

- Salience-weighting changed only **4 of 24 frames** on *both* videos (83% identical).
- The resulting digests were **statistically indistinguishable**: paywalls 23 vs 22
  numeric tokens; onboarding 45 vs 49; anchored specifics (Opal 7→17%, Slopes 25%,
  Headspace 10%, Grammarly 20%, Brian Chesky, 23%/7%…) captured **equally** by both.

**Why:** detail-max already lands on anchored slides — a video shows a stat slide *while*
the narrator discusses it, so the most-detailed frame per window is usually already the
right one. Salience alignment adds nothing measurable. → **Do not build approach A.**
Approach C (two-pass LLM claim-anchoring) is untested and costs 2× LLM calls — not worth
it given A's flat result.

## The actual finding — the lever is candidate-pool DENSITY, not alignment

The experiment fed both selectors a **dense** interval pool (every 6s). With it, *both*
selections captured the anchored stats that P2's *shipped* scene-detection pool missed in
the earlier E2E. So the cheap, real improvement to the shipped feature is: in `frames.js`,
**supplement scene detection with a denser interval candidate pool** before
`selectDetailFrames` runs (always also sample ~1 frame / 6–8s, merge, then bin-select by
detail). No salience, no transcript plumbing, no prompt change — a small `frames.js` tweak,
worth its own quick before/after check. This is the recommended follow-up instead of P3.

## The gap P3 closes

P2 made selection pick the most **visually detailed** frames (blanks and talking-heads
dropped). But the E2E proved *detailed ≠ valuable*: on the paywalls video the 24
selected frames were mostly **incidental B-roll** — paywall screenshots the video
flashes by (Arrow Gold $29.99, Lóvi Premium…). The digest correctly ignored them,
because **the narrator wasn't talking about them.**

The frames that actually paid off were the ones whose on-screen content **anchored to a
point the narration was making** — the spin-wheel `75%` (narrator discusses spin-wheel
tactics), the Blinkist stats in the manual test (narrator discusses Blinkist's
redesign). A visual-detail score can't tell an anchored stat slide from busy B-roll;
they're all just "detailed slides."

**P3 = prefer frames whose timestamp lands on substantive, specific narration, and tell
the digest which narration each frame belongs to.**

## The enabler (already present, currently discarded)

Both sides already carry timestamps — the join just isn't used:
- **Frames:** `frames.js extractFrames` returns `[{ path, offsetSec }]` (P1 built the
  `showinfo`/`pts_time` timestamping).
- **Transcript:** `transcript.js fetchTranscript` returns `[{ text, offset }]` (seconds).

But the **timestamps are thrown away before the digest**: the frontend flattens
segments to plain text via `buildPlainTranscript()` (drops offsets), and
`POST /api/digest` receives only `text`. So today the digest sees a loose pile of
frames and a timeless wall of text. P3's precondition is keeping the segment offsets
alive down to the frame-selection + digest step.

## Approaches (cheapest first)

### A. Salience-weighted co-selection *(recommended for P3 v1)*
Re-rank candidate frames by **both** visual detail **and** narration salience at their
timestamp, instead of detail alone.
- **Salience of a transcript window** near a frame's `offsetSec` = density of *specific*
  tokens: numbers, currency, `%`, dates, and capitalized proper nouns/multi-word names
  (a cheap regex/heuristic — no LLM). A frame sitting on "…Blinkist redesigned its trial,
  result +23%…" scores high; one over "…so anyway, you know, the thing is…" scores low.
- **Combined score** = `norm(jpegSize) * (1 + w * salience)` — keep the P2 detail proxy,
  boost frames aligned to specific narration. Per-temporal-bin max by combined score
  (reuse P2's `pickByBins`).
- Zero extra LLM cost; pure heuristic over data we already have.

### B. Per-frame narration context in the prompt *(pairs with A, recommended)*
Instead of "here are 24 frames," tell the digest **when** each frame was on screen and
what was being said: `frame-007.jpg (~4:32) — on screen while: "<the transcript window
around 4:32>"`. This lets the model fuse a frame's numbers into the *right* point rather
than guessing where a loose frame belongs. Changes `buildFramePromptBlock` to emit a
timestamped, context-annotated frame list. Cost: a bit more prompt text.

### C. Two-pass claim-anchoring *(P3.5 — only if A+B underperform)*
A cheap first LLM pass extracts the video's key claims + their transcript timestamps;
then select and read *only* the frames at those timestamps to ground them. Most powerful,
most expensive (2× LLM calls, more orchestration). Defer until A+B are measured.

**Recommendation:** ship **A + B** together for P3 v1 — they share the one precondition
(segment offsets reaching the frame path), add no LLM cost, and directly target the gap.

## Data-flow change (the one real plumbing cost)

Segment offsets must reach frame selection + the digest. Two options:
1. **Frontend sends `segments`** (it already holds `lastSegments`) alongside `text` on
   `POST /api/digest`. No redundant fetch; but a bigger request body (fine —
   local/desktop only, and the web cap doesn't apply since frames are web-disabled).
2. **Server re-fetches** `fetchTranscript(videoId)` for its offsets when
   `includeVisuals`. No client change; costs one keyless transcript re-fetch.

Lean **option 1** (client already has the segments; avoids a duplicate fetch). Guard:
segments are only needed on the frames path, so this stays inside the `wantVisuals`
branch and never touches the plain text-digest contract.

## Code touchpoints

- **`frames.js`** — `extractFrames` should return a **larger candidate pool** (e.g. keep
  the top ~2–3× `maxFrames` by detail, or return all candidates with `{path, offsetSec,
  size}`) so the aligner can re-rank by salience instead of the visual-only cap deciding
  first. Move the final `maxFrames` cap into the aligner. `selectDetailFrames` stays as
  the fallback when no segments are supplied.
- **new `alignFrames.js`** (or a function in `digest.js`) — `selectAnchoredFrames(candidates,
  segments, maxFrames, opts)`: compute per-frame salience from the nearest segment
  window, combine with detail, per-bin max, cap, and (for approach B) return each kept
  frame with its `contextText`. Pure + unit-testable (feed synthetic candidates+segments).
- **`digest.js`** — `generateDigest` accepts `opts.frames.items[].contextText` and
  `buildFramePromptBlock` emits the timestamped, context-annotated list (approach B).
- **`server.js` `/api/digest`** — accept `segments`; when `wantVisuals`, pass them into
  the align step; keep the best-effort/degrade + `finally` cleanup exactly as P2.
- **`public/index.html`** — send `segments` (from `lastSegments`) in the `includeVisuals`
  request. Local/desktop only, as today.

## Faithfulness

Keep P2's **numbers-encouraged / names-strict** clamp verbatim — approach B *reduces*
fabrication risk (it hands the model the real narration context, so it has less reason to
invent). Do not relax the names rule; the alignment context is grounding, not licence.

## The experiment to run FIRST (gate)

Mirror the P1/P2 method — measure before building:
1. Take 2 videos: the Mobbin paywalls teardown (have it) + one lecture/slide talk where
   visuals carry info (e.g. a 3Blue1Brown-style or conference-slides video).
2. For each: run the P2 digest (detail-only selection) vs a **prototype** salience-weighted
   selection (approach A, scripted in scratchpad — no need to wire the server yet).
3. Compare: does salience-weighting surface the **narration-anchored** stat slides
   (Opal 7→17%, Slopes 25%, Blinkist +23/-55) that P2 missed? Count frame-only specifics
   captured, and check the selected frames' contents (ground-truth via an image agent, as
   in the prior tests).
4. **Ship P3 only if salience-weighting measurably captures more anchored specifics than
   detail-only.** If detail-only is already close, P3 isn't worth the plumbing.

## Non-goals / risks

- **Not web.** Same as frames overall.
- **Timestamp skew.** A frame's `offsetSec` (scene-onset) may lead/lag the narration that
  references it by a few seconds; the salience window must be generous (±~8s), and the
  onset-vs-mention gap is a known imprecision to measure, not assume away.
- **Salience heuristic is crude.** Regex specificity ≠ semantic importance; it will miss
  claims phrased plainly and over-weight number-dense filler. Approach C (LLM claim
  extraction) is the fallback if the heuristic is too noisy — but it's more cost, so gate
  on the experiment.
- **Diminishing returns.** P1+P2 already deliver measured value (≈2× concrete figures).
  P3 is the "surface the *best* on-screen facts, reliably" upgrade — real, but not a
  prerequisite for the feature being useful.

## Open decisions

1. Segments to the frame path via **frontend-sends** (assumed) vs **server-refetch**.
2. Salience weight `w` and window size (±8s assumed) — tune in the experiment.
3. Candidate-pool size returned by `extractFrames` (2× vs 3× `maxFrames`).
4. Whether approach B's per-frame context is worth the added prompt length on its own, or
   only alongside A.
